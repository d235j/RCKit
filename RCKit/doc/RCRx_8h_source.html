00007 /// \mainpage RCKit library for Arduino
00008 ///
00009 /// This is the Arduino RCKit library.
00010 /// 
00011 /// RCKit provides a kit of software objects that make it easy to build an
00012 /// RCOIP  (Remote Control Over IP) receiver on Arduino. RCOIP protocol is used to
00013 /// carry remote control commands from a transmitter to a receiver over an IP
00014 /// transport such as Ethernet or Wi-Fi.
00015 ///
00016 /// As such it can be used to build Remote Controlled (RC) vehicles and devices of various kinds, along with
00017 /// matching transmitters. Large numbers of channels can be supported, along with a back-channel 
00018 /// which sends data from the Receiver to the Transmitter. 
00019 /// A compatible iPhone transmitter app is also available.
00020 /// 
00021 /// The transmitter could be a handheld Wi-Fi enabled device such as an iPhone, iPad or Arduino based device. 
00022 /// Or it could be an interactive program running on a desktop computer. The RCTx iPhone transmitter app is 
00023 /// available on the Apple App Store. It provides a simple RCOIP transmitter equipped with 2 joysticks and
00024 /// a number of switches.
00025 ///
00026 /// The receiver (using this RCKit software) could be a remote controlled car, 
00027 /// plane, helicopter or some other device.
00028 ///
00029 /// The RCOIP protocol is a 2-way protocol that defines UDP messages between 
00030 /// the RCOIP transmitter and receiver. 
00031 /// This effectively makes the transmitter a UDP client and the receiver a UDP server. 
00032 /// Messages sent from the transmitter to the receiver include setting analog outputs, 
00033 /// and messages from the receiver to the transmitter include receiver status messages.
00034 ///
00035 /// RCTx, RCKit and the RCOIP protocol offer the following advantages over conventional Remote Control:
00036 /// \li Large numbers of channels (> 100) not just 5 or 6, including analog, digital, text, data etc.
00037 /// \li Back channel for telemetry (voltages, signal strengths, GPS position etc)
00038 /// \li Failsafe modes
00039 /// \li Programmable, configurable and extensible
00040 /// \li Works with a variety of types of vehicle and remote devices
00041 /// \li Hackable
00042 /// \li Opportunity to tightly integrate remote control and autonomous vehicle software control
00043 /// \li Works with a variety of IP transports Wi-Fi (ad-hoc or infrastructure), Wired etc.
00044 ///
00045 /// Videos explaining how it works and what you can do with it can be found at:
00046 /// \li http://www.youtube.com/watch?v=SKwY-CcotRY
00047 /// \li http://www.youtube.com/watch?v=i0ZogFmytPI
00048 ///
00049 /// There is also a video showing how RCKit can be used with the IRrc library at
00050 /// http://www.open.com.au/mikem/arduino/IRrc to control a 3 channel infra-red model helicopter at:
00051 /// \li http://www.youtube.com/watch?v=lzRpyqnD6_M

00066 /// \par RCRx
00067 ///
00068 /// RCRx class is an RCOIP receiver. It works with Arduino and a range of communications hardware like 
00069 /// WiShield and Ethernet to receiver commands from an
00070 /// RCOIP compliant Transmitter (such as the RCTx iPhone app available on the Apple App Store).
00071 ///
00072 /// When the RCRx object is constructed, it must be given an array of Setter objects. 
00073 /// When a RCOIPv1CmdSetAnalogChannels message is received by RCRx, The Setter corresponding to each 
00074 /// Channel value in the RCOIPv1CmdSetAnalogChannels command will be passed to the 
00075 /// respective Setter in the analogOutputs array. 
00076 /// The Arduino software is expected to be configured so that each analogOutput is connected to a Setter that
00077 /// will implement the received value, perhaps by setting an analog or digital output on an Arduino pin. 
00078 /// Setter objects are provided to achieve this. See below.
00079 ///
00080 /// RCRx will automatically respond to received commands with RCOIPv1ReplyReceiverStatus messages 
00081 /// as appropriate to the configured timeouts etc. See the RCOIP protocol document for more details.
00082 /// 
00083 /// \par Setters
00084 ///
00085 /// Setter classes are objects that receive a value, maybe transform it and then do something with it. Typically
00086 /// they set an output pin according to the received input value, but almost any other transformation, 
00087 /// output or communication can be imagined. 
00088 ///
00089 /// The following output Setter objects are provided with RCKit. They can be used to translate 
00090 /// receiver channel values into physical outputs:
00091 /// \li AnalogSetter
00092 /// \li DigitalSetter
00093 /// \li ServoSetter
00094 /// \li AccelStepperPositionSetter
00095 /// \li AccelStepperSpeedSetter
00096 /// \li HBridgeSetter
00097 /// \li DifferentialSetter
00098 ///
00099 /// The following transformation Setter objects are provided with RCKit. They can be used to 
00100 /// transform receiver channel values before being given to one of the output Setter classes above:
00101 /// \li Inverter
00102 /// \li Limiter
00103 /// \li Linear
00104 /// 
00105 /// \par Example Sketches
00106 ///
00107 /// Several example Arduino sketches are included, including a regression 
00108 /// test suite and a sample complete 5 channel receiver with Servo outputs for WiFi and Ethernet.
00109 ///
00110 /// \par Transceivers
00111 ///
00112 /// Version 2.0 of this library added the concept of a Transceiver, in order to be able to support 
00113 /// multiple type of communications transport.
00114 /// A Transceiver is an object responsible for communicating with an RCOIP transmitter. Several types of 
00115 /// Transceiver are supported by the standard RCRx library:
00116 /// - WiShield WiFi shield or Yellowjacket, using the WiShield library.
00117 /// - Ethernet shield, or EtherTen, using the standard Arduino Ethernet library.
00118 /// You can define your own subclass of the Transceiver class to implement your own transports 
00119 /// (if you do so, consider contributing it back to the RCRx project).
00120 /// The main RCRx reciver object must be told where it Transceiver object is, using setTransceiver(). 
00121 /// Thereafter it will use that 
00122 /// Transceiver to receive RCOIP requests and to send replies back to the transmitter.
00123 ///

00184 /// \par WiShield Library Configuration
00185 ///
00186 /// Requires the Asynclabs WiShield library to be installed, even if you are not using
00187 /// the WiShieldTransceiver. See http://asynclabs.com/wiki/index.php?title=WiShield_library
00188 /// Install the  WiShield library in the libraries directory of your arduino IDE installation, 
00189 /// then follow the configuration steps below:
00190 ///
00191 /// Support of RSSI (receiver signal strength indicator) requires mods to WiShield library g2100.c as per 
00192 /// http://asynclabs.com/forums/viewtopic.php?f=10&t=385&start=0. You dont have to add this but its a 
00193 /// good feature. It is included in the prebuilt WiShield library mentioned below.
00194 ///
00195 /// Correct operation of the WiShield requires you to set the jumper on the WiShield to INT0 or DIG8 to select 
00196 /// the arduino pin to use for WiShield interrupts, and also to make sure it agrees with the settings of 
00197 /// USE_DIG0_INTR or  USE_DIG8_INTR in spi.h in the WiShield library (which defaults to 
00198 /// using Arduino digital pin 2, and which means setting the WiShield jumper to INT0 setting). Yes, the naming
00199 /// conventions are inconsistent :-(. In summary:
00200 /// \code
00201 ///  WiShield jumper   spi.h          Arduino
00202 ///       INT0        USE_DIG0_INTR  Digital pin 2
00203 ///       D8          USE_DIG8_INTR  Digital pin 8
00204 /// \endcode
00205 ///
00206 /// For YellowJacket (which has no jumper), leave it as USE_DIG0_INTR.
00207 ///
00208 /// In order for WiShield library to support UDP (as needed by this module), 
00209 /// you MUST set UIP_CONF_UDP to 1 in uip-conf.h. This is an unfortunate but necessary requirement, 
00210 /// otherwise UDP support will not be compiled into the WiShield library. 
00211 /// Further, you must edit apps-conf.h and make sure the only APP_* defined is APP_UDPAPP. 
00212 /// Failure to do this will cause compile errors. 
00213 ///
00214 /// A modified version of the WiShield library already 
00215 /// modified and configured for use with RCKit (including RSSI support) is available at 
00216 /// http://www.open.com.au/mikem/arduino/WiShield-v1.3.0-0-mikem-RCKit.zip
00217 ///
00218 /// WiShield will work with Arduino Mega, but with difficulty. The problem is that with the Mega, the SPI 
00219 /// pins that are required for interface with WiShield come out on different pins to the smaller form 
00220 /// factor arduinos like Diecimila and Duemilanove. So, to make the Mega work with the WiShield, you 
00221 /// have to reroute the SPI pin to different Arduino pins, as per 
00222 /// http://asynclabs.com/forums/viewtopic.php?f=13&t=19&hilit=mega&start=10
00223 /// 
00224 /// By default, WiShieldTransceiver is configured as an Ad-Hoc Wi-Fi network with SSID of 'RCArduino' 
00225 /// By default, the network is Open (ie no encryption) and
00226 /// It is configured with a static IP address of 169.254.1.100.
00227 /// These setting can be changed by editing WiShieldTransceiver.cpp in the RCKit distribution.

00282 /// \brief Remote Control Receiver module for RCOIP protocol on Arduino.
00283 ///
00284 /// \par Overview
00285 /// This class implements a receiver for RCOIP (Remote Control Over IP). It starts and manages a 
00286 /// WiFi receiver, which receives UDP messages containing RCOIP commands such as remote control channel 
00287 /// values. When channel setting commands are received they are translated into output values which are sent 
00288 /// to Setter objects to control the phycical output devices and pins on the Arduino. 
00289 /// Supports WiShield 1.0 etc.
00290 ///
00291 /// \par Outputs
00292 /// RCRx maps RCOIP channels to physical output devices through the analogOutputs array. This is an array 
00293 /// of Setter objects, one for each physical output to be controlled by RCRx. Whenever a RCOIP message 
00294 /// is received with new channel settings, the input() function of each Setter with new data will be 
00295 /// called. This will cause each Setter to set its physical output in response to the remote control
00296 /// data received by an RCOIP. 
00297 ///
00298 /// \par Failsafe
00299 /// RCRx supports failsafe behaviour if a connection to the transmitter is lost.
00300 /// RCRx monitors the time of each received RCOIP request. If no request is received for more than 
00301 /// failInterval milliseconds, it will be considered as disconnected, and the failsafe() function of 
00302 /// every Setter will be called, allowing each Setter to adopt its failsafe configuration 
00303 /// (eg throttle to 0 etc). This allows remote control vehicles to fail safe if the transmitter 
00304 /// fails or goes out of range.
